# Session Log: 2026-01-28 - AMM Bug Fixes

## Summary

Debugged and fixed critical bugs in pre-generated AMM ledger objects that were causing `temBAD_AMOUNT` errors on two-asset AMMDeposit transactions.

## Bugs Fixed

### 1. AMM Account Derivation (`src/generate_ledger/indices.py`)

**Problem**: AMM account address was being calculated incorrectly, causing rippled to return "Account malformed" on `amm_info` queries.

**Root Cause**: Extra SHA512Half step in the derivation formula.

```python
# WRONG (before)
preimage = i_bytes + parent_hash + amm_index_bytes
hash_result = _sha512_half(preimage)  # This step shouldn't exist!
account_id = _ripesha(hash_result)

# CORRECT (after)
preimage = i_bytes + parent_hash + amm_index_bytes
account_id = _ripesha(preimage)  # Direct RIPESHA, no SHA512Half
```

**Also fixed**: Changed `i_bytes` from little-endian (`<H`) to big-endian (`>H`) to match rippled's network byte order.

### 2. lsfAMMNode Flag (`src/generate_ledger/amm.py`)

**Problem**: Two-asset AMMDeposit transactions failed with `temBAD_AMOUNT`, while single-asset XRP deposits worked fine.

**Root Cause**: Wrong flag value for AMM trustlines.

```python
# WRONG (before)
LSF_AMM_NODE = 0x02000000  # This is actually lsfLowDeepFreeze!

# CORRECT (after)
LSF_AMM_NODE = 0x01000000  # This is the real lsfAMMNode
```

**Impact**: The wrong flag (0x02000000 = lsfLowDeepFreeze) was freezing the asset trustlines between the AMM account and token issuers. Frozen trustlines cannot transfer tokens, which blocked any USD deposits to the AMM.

## Debugging Journey

1. Initial symptom: `temBAD_AMOUNT` on AMMDeposit with tfTwoAsset flag
2. Checked `amm_info` - got "Account malformed" error
3. Fixed AMM account derivation (removed SHA512Half)
4. After fix, `amm_info` worked but deposits still failed
5. Bob's single-asset XRP deposit succeeded (proved AMM was valid)
6. Two-asset deposits still got `temBAD_AMOUNT`
7. Created fresh AMM via AMMCreate - two-asset deposits worked!
8. Compared trustlines: pre-generated had `deep_freeze_peer: true`
9. Traced flag value: 0x02000000 is lsfLowDeepFreeze, not lsfAMMNode
10. Fixed flag to 0x01000000

## Lessons Learned

1. XRPL flag values are easy to get wrong - always verify against rippled source
2. `temBAD_AMOUNT` can be misleading - the actual cause was a frozen trustline
3. Single-asset deposits can work while two-asset fails (different code paths)
4. Comparing pre-generated objects with AMMCreate results is a good debugging technique
5. `account_lines` RPC reveals trustline flags that aren't obvious in ledger.json
